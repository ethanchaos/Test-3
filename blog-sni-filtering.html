How SNI Is Used to Filter & Block Traffic

SNI-based filtering is how modern censorship systems block HTTPS websites without decrypting your traffic. Here's the full technical picture — how it works, where it's deployed, and what countermeasures exist.

📖 ~6,500 words⏱️ 26 min read🏷️ Advanced · Research

Why SNI Filtering Exists

Before HTTPS became universal, internet filtering was relatively straightforward: HTTP requests were plaintext, so a firewall could read the destination URL directly and apply filtering rules. Blocking http://example.com/blocked-page was as simple as matching the URL in the packet data.

The global push to HTTPS changed this. By 2023, over 95% of web traffic was HTTPS — encrypted at the TLS layer. The content of requests became unreadable to network observers. This threatened to make content-based filtering impossible.

SNI provided the solution — or rather, the loophole that preserved filtering capability in the HTTPS era. Because the SNI field in the TLS Client Hello is sent before encryption is established, it remains visible to any observer on the network path. A filtering system that reads SNI can identify the destination domain of any HTTPS connection — and apply blocking rules based on that domain — without ever decrypting the traffic.

How SNI Filtering Works — The Technical Pipeline

SNI-based filtering is typically implemented as part of a Deep Packet Inspection (DPI) system deployed at strategic points in the network — typically at IXPs (Internet Exchange Points), at major backbone routers, or at ISP border routers where international traffic enters and leaves the network.

  1. Packet capture: Network hardware with DPI capabilities examines packets in real time as they flow through the network.
  2. TLS handshake detection: The DPI system identifies packets that are part of TLS handshakes — specifically looking for the Client Hello message (identified by its record type byte and handshake type byte).
  3. SNI extraction: The system parses the TLS extension list in the Client Hello to find the SNI extension (extension type 0x0000) and extracts the hostname.
  4. Blocklist lookup: The extracted hostname is checked against a blocklist of forbidden domains.
  5. Action: If the domain is blocked, the system either drops the packets (connection timeout for the user), sends a TCP RST (connection reset), or redirects to a block page.

Where SNI Filtering Is Deployed

China — Great Firewall

China's Great Firewall has used SNI filtering since at least 2012, making it one of the earliest large-scale deployments. The GFW's SNI filtering system is notable for its sophistication: it doesn't just match exact domain names but can also match subdomains and apply filtering based on more complex rules. The GFW also combines SNI filtering with active probing — when it detects a potentially blocked connection, it probes the server to verify it's running a blocked service before committing to a block.

Russia

Russia's TSPU (Technical Means for Countering Threats) system, deployed under the "sovereign internet" law of 2019, includes SNI-based filtering capabilities. The system is managed by Roskomnadzor and can be updated centrally to add domains to the block list. Russia's SNI filtering is notably less consistent than China's — enforcement varies by ISP and region.

Iran

Iran operates one of the most aggressive SNI filtering systems outside China. During periods of political unrest (most notably the 2022 Mahsa Amini protests), Iran's filtering system was used to block Instagram, WhatsApp, and other platforms at the SNI level while keeping connectivity to other services intact.

Corporate and Educational Firewalls

SNI filtering is not exclusively a government tool. Many corporate firewall products (Palo Alto, Cisco, Fortinet, Check Point) implement SNI-based filtering as a core feature for content policy enforcement. School networks routinely use SNI filtering to block social media, gaming sites, and other non-educational content. These systems are built on exactly the same technical principles as state-level censorship — the difference is who controls the block list and for what purpose.

SNI Filtering Detection Tools

Several tools and platforms can help identify whether SNI filtering is in use on a given network:

  • OONI Probe — Includes tests specifically for SNI-based blocking. The OONI "SNI blocking" test sends TLS Client Hello packets with specific SNI values to test servers and measures whether the connections are blocked. Results are published in OONI Explorer.
  • Netalyzr — Network analysis tool that can detect various forms of traffic interference including SNI filtering.
  • Chaos SNI Finder — Ethan Chaos's tool, which among other functions tests which SNI values successfully pass through a given network — effectively mapping which domains are blocked at the SNI level and which aren't.

Countermeasures Against SNI Filtering

Encrypted Client Hello (ECH)

The most promising technical countermeasure is Encrypted Client Hello (ECH), which encrypts the entire Client Hello message — including the SNI field — making it unreadable to network observers. With ECH deployed, SNI-based filtering becomes technically impossible without blocking all HTTPS traffic.

ECH is defined in RFC 9849 and is increasingly deployed: Cloudflare supports it for all domains on their CDN, Firefox enables it by default with DoH, and Chrome has supported it since version 117. The challenge is that ECH requires HTTPS DNS records (to distribute the public key used for encryption), and many DNS providers don't yet support these records.

In response to ECH adoption, some filtering systems have started blocking connections where ECH is detected — effectively requiring traffic to use plaintext SNI or be dropped. This is the same pattern we saw with ESNI and represents the ongoing arms race between privacy technology and filtering infrastructure.

SNI Padding / Null SNI

Some circumvention tools experiment with sending a null or empty SNI field, or padding the SNI with extra characters. Some filtering systems only match on complete, valid SNI values and ignore malformed or empty ones. This is a fragile technique — sophisticated filters handle these cases — but it can work against less careful implementations.

VPN and Tunneling

The most reliable countermeasure is using a VPN or tunnel that wraps your traffic before the outer connection's SNI is visible. If your VPN connection itself uses an SNI value that isn't blocked (or uses ECH, or spoofs a whitelisted SNI), the blocking system only sees the VPN connection's outer SNI — not the SNI of the sites you're actually accessing through the tunnel.

ESNI (Deprecated but Instructive)

Before ECH, Encrypted SNI (ESNI) was deployed by Cloudflare as an experimental solution. It encrypted just the SNI extension while leaving the rest of the Client Hello visible. Several countries (China, Russia) responded by blocking all traffic with the ESNI extension — an example of blocking the privacy mechanism itself rather than its content. ECH was designed to be more resistant to this kind of blocking by making it harder to distinguish ECH traffic from regular TLS traffic.

Designing Configs That Work Around SNI Filtering

For config builders working in environments with SNI filtering, the key design principles are:

  1. Use an SNI value that isn't blocked. The outer SNI (the one the filtering system sees) must be a domain that isn't on the block list. This is why zero-rated domains are often used as SNI values in carrier configs — they're typically whitelisted rather than blocked.
  2. Ensure the actual traffic is encrypted and tunneled. The filtering system should never see the SNI of the sites you're actually trying to access — those connections happen inside the tunnel, where the outer network can't inspect them.
  3. Test, iterate, and update. Block lists and filtering rules change constantly. A config that works today may not work tomorrow. This is why community-maintained config hubs like the Chaos Configurations Config Hub are valuable — they provide continuously updated resources.
  4. Consider ECH where available. Where ECH is supported by both the server and the network, it provides the most robust protection against SNI-based filtering.
