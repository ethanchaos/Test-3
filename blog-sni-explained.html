How SNI Works

Server Name Indication is one of the most important — and least understood — pieces of modern internet infrastructure. It's how HTTPS servers know which certificate to serve, and it's also how ISPs, governments, and firewalls know which sites you're connecting to, even over encrypted connections.

📖 ~6,000 words ⏱️ 25 min read 🏷️ Intermediate · Advanced

The Problem SNI Solves

To understand why Server Name Indication (SNI) exists, you first need to understand a fundamental challenge in web hosting: multiple websites sharing a single IP address.

In the early days of the web, every website needed its own dedicated IP address if it wanted to support HTTPS. This worked fine when there were relatively few websites, but as the internet scaled, it became impractical. IPv4 addresses — the most widely used addressing scheme — are limited, and the pool of available addresses was exhausted years ago. Requiring a dedicated IP address for every HTTPS website would have been catastrophically expensive and wasteful.

For regular HTTP (unencrypted) websites, the problem was solved easily: the browser includes the domain name in the HTTP Host header, which the server reads to figure out which website to serve. A single server can host thousands of websites at one IP address because it reads the Host header in each request to know which one to respond with.

But HTTPS made this harder. HTTPS encrypts the HTTP request — including the Host header — inside a TLS connection. And here's the catch: the TLS handshake happens before the HTTP conversation begins. Before the server can read the Host header (to know which website you want), it has to complete the TLS handshake — but to complete the handshake, it needs to present a certificate. To present the right certificate, it needs to know which website you're connecting to. This is a chicken-and-egg problem.

⚡

The Core Paradox

Without SNI: The server can't know which certificate to serve until it reads the Host header. But it can't read the Host header until it completes the TLS handshake. And it can't complete the TLS handshake without serving a certificate. Classic deadlock.

The solution — Server Name Indication — was published in RFC 3546 in June 2003. SNI adds the requested domain name as an extension to the TLS Client Hello message — the very first message in the TLS handshake, sent before any encryption is established. The server reads the SNI field, selects the appropriate certificate, and the handshake proceeds normally.

SNI in the TLS Handshake — Step by Step

The TLS handshake is the process by which two parties establish a secure encrypted connection. Here is exactly how it works, with SNI included:

  1. Client Hello (includes SNI)
    Your browser sends the initial Client Hello message. This message includes:
    • The TLS version(s) it supports
    • A list of cipher suites it supports (encryption algorithms)
    • A randomly generated number (client random)
    • The SNI extension — the hostname of the site you're connecting to
    At this point, no encryption has been established yet. The Client Hello — including the SNI field — is sent in plaintext over the network. This is the critical vulnerability that makes SNI-based filtering possible.
  2. Server Hello + Certificate
    The server reads the SNI field, selects the correct SSL/TLS certificate for that domain, and responds with its Server Hello message including the selected certificate.
  3. Certificate Verification
    Your browser verifies that the certificate is valid — signed by a trusted Certificate Authority, not expired, and matching the domain name.
  4. Key Exchange
    Both sides perform a cryptographic key exchange (using ECDHE or similar) to establish a shared session key without transmitting it over the network.
  5. Encrypted Session Begins
    From this point on, all communication is encrypted with the shared session key. The HTTP request (including the Host header) is now sent encrypted inside the established TLS session.

Why SNI Is Sent in Plaintext — and Why This Matters

The fact that SNI is sent in plaintext before encryption is established was not an oversight — it was a necessary design choice at the time. The server needs to know which certificate to present before it can establish encryption, and there was no mechanism in TLS 1.2 and earlier to encrypt the handshake itself.

This means that anyone who can observe your network traffic can see which domains you're connecting to, even when you're using HTTPS. This includes:

  • Your Internet Service Provider (ISP)
  • Network administrators at your workplace or school
  • Government surveillance systems monitoring internet traffic
  • Anyone running a packet sniffer on the same network as you (e.g., on a public Wi-Fi hotspot)

They can't see what you're doing on those sites — the content of your HTTP requests and responses is encrypted. But they can see which domains you're connecting to and when. For many privacy and censorship scenarios, this is enough information to be problematic.

⚠️

Real-World Impact

China's Great Firewall, Russia's TSPU system, and corporate content filters all use SNI inspection to block HTTPS websites without needing to decrypt traffic. Just reading the SNI field in the Client Hello is sufficient to identify and block connections to targeted domains.

SNI in VPN Configurations — Why It's Used

SNI plays a fascinating and counterintuitive role in VPN configurations, particularly in the type of carrier-bypass configs used with apps like HTTP Injector. Understanding this requires thinking about how carrier firewalls work.

Many mobile carriers — particularly in developing markets — implement billing and access control at the network level. When you run out of data, they don't just cut off your connection entirely — they implement rules that block most traffic while allowing certain "free" or zero-rated domains to pass through. These zero-rated domains are whitelisted at the carrier level.

When your device makes an HTTPS connection to a zero-rated domain, the carrier's network sees the SNI field in the Client Hello and recognizes the domain as whitelisted — so it allows the connection to proceed without deducting from your data balance.

This is where the cleverness of SNI-based VPN configs comes in. A properly crafted HTTP Injector or HA Tunnel Plus configuration can use an SNI value that matches a whitelisted domain while actually routing traffic to a different server (the VPN server). The carrier's firewall sees the SNI and thinks the connection is going to a whitelisted domain — so it allows it. The actual traffic, however, is going somewhere else entirely.

This technique is called SNI spoofing or SNI manipulation. It works because the SNI field, while plaintext, is typically only used by the carrier for the whitelist check — not for verifying the final destination of the traffic.

Note: The effectiveness of SNI-based configs varies significantly by carrier and region, and as carriers become more sophisticated in their DPI implementations, SNI spoofing alone may not be sufficient. More advanced configs combine SNI manipulation with additional header injection and routing tricks.

Encrypted Client Hello — The Future of SNI Privacy

The networking community has long recognized that plaintext SNI is a significant privacy vulnerability. The solution — Encrypted Client Hello (ECH) — has been in development for several years and was formally published as RFC 9849 in March 2026.

ECH works by encrypting the entire Client Hello message — including the SNI extension — using a public key that the client obtains via DNS before making the connection. With ECH, anyone monitoring your network traffic sees an encrypted Client Hello with a generic "outer" server name (typically the CDN or DNS provider), but the actual target domain is hidden inside the encrypted payload.

How ECH Changes the Picture

ECH fundamentally breaks SNI-based content filtering. If the SNI field is encrypted, firewalls and ISPs can no longer use it to identify which domain you're connecting to. This has significant implications:

  • For privacy: ECH is a major improvement. Your ISP and network observers can no longer see which HTTPS domains you're visiting.
  • For censorship circumvention: SNI-based blocking becomes ineffective. Censors would have to block all HTTPS traffic to a CDN (causing massive collateral damage) or switch to other filtering methods.
  • For content filtering tools: Enterprise and school network filters that rely on SNI inspection will need to adapt — likely by requiring traffic to pass through explicit proxies or using DNS-based filtering instead.
  • For carrier-based zero-rating configs: SNI spoofing techniques will become less reliable as networks upgrade to ECH-aware infrastructure. New approaches will be needed.

ECH is currently supported in Firefox (with DoH enabled), Chrome (in recent versions), and Cloudflare's network. Adoption is growing but not yet universal. The transition to ECH represents a significant shift in the privacy and censorship landscape — and it's something we're tracking closely at Chaos Configurations.

Domain Fronting — A Related Technique

Domain fronting is a technique that exploits the distinction between the SNI field (which the carrier/firewall sees) and the HTTP Host header (which the actual server sees, after TLS decryption). The idea is to put a benign, whitelisted domain in the SNI field while putting the actual target domain in the encrypted HTTP Host header.

For domain fronting to work, both the fronting domain (the one in SNI) and the target domain must be hosted on the same CDN or server infrastructure. If they're on the same CDN, the CDN's edge servers accept the connection (seeing the whitelisted SNI), then route the request to the actual backend server based on the Host header.

Domain fronting was used for censorship circumvention by tools like Signal and Tor's meek transport. It became less viable when major CDN providers (Google, Amazon AWS, Cloudflare) updated their configurations to reject connections where the SNI domain doesn't match the HTTP Host header. But it remains conceptually important for understanding the relationship between SNI and routing.

ESNI and Government Blocking

The predecessor to ECH was Encrypted SNI (ESNI), which was an experimental protocol that encrypted just the SNI field (rather than the entire Client Hello). ESNI was deployed by Cloudflare and supported in Firefox from 2018 to 2021.

Interestingly, some governments chose to block ESNI traffic entirely rather than adapt their filtering systems. China's Great Firewall began blocking ESNI traffic in August 2020. Russian ISPs started blocking it in October 2020. The rationale was straightforward: if you can't see the SNI, you can't filter by domain name — so block the encryption protocol itself to force connections to use plaintext SNI.

This represents a recurring pattern in the censorship-circumvention arms race: as privacy-enhancing technologies improve, censors respond by blocking the privacy enhancement rather than its contents. The same dynamic played out with Tor (blocked in many countries), VPNs (blocked in China, Russia, UAE), and ESNI. ECH will face the same challenge.
