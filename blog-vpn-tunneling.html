VPN Tunneling Deep Dive

How HTTP, SSL, and SSH tunneling work at the protocol level — and how traffic is routed, encapsulated, and disguised to traverse restricted networks. The most complete explanation of VPN tunneling mechanics on the web.

📖 ~7,500 words ⏱️ 30 min read 🏷️ Advanced

What Is Tunneling?

A network tunnel is a communication pathway created by encapsulating one network protocol inside another. The outer protocol carries the inner protocol's data as its payload — the same way a physical tunnel allows a road to pass through a mountain. From the outside, you see the mountain (the outer protocol). Inside the tunnel, there's a road (your actual traffic).

Tunneling is fundamental to how VPNs work. When you connect to a VPN, all of your network traffic — HTTP requests, DNS queries, video streams, everything — is encapsulated inside the VPN protocol's packets. The VPN server unwraps those packets, forwards the original traffic to its destination, receives the response, re-encapsulates it, and sends it back to you.

But tunneling goes beyond basic VPNs. In the world of network restriction circumvention, tunneling is used not just to encrypt traffic but to disguise what type of traffic it is. A tunnel that makes VPN traffic look like ordinary HTTPS browsing is far harder to detect and block than one that uses obvious VPN signatures.

HTTP Tunneling

HTTP tunneling is the technique of sending non-HTTP traffic through an HTTP connection. The most common method uses the HTTP CONNECT method, which was originally designed to allow HTTP proxies to relay HTTPS connections.

The HTTP CONNECT Method

Here's how HTTP CONNECT works in its normal use: when your browser wants to make an HTTPS connection through an HTTP proxy, it can't just send the encrypted HTTPS request to the proxy (the proxy wouldn't know what to do with it). Instead, it first sends an HTTP CONNECT request to the proxy:

CONNECT target-server.com:443 HTTP/1.1
Host: target-server.com:443
Proxy-Authorization: Basic base64encoded_credentials

The proxy, if it allows this connection, responds with:

HTTP/1.1 200 Connection Established

From this point, the TCP connection between the browser and the proxy is now a raw tunnel to target-server.com:443. The browser sends its HTTPS Client Hello, which passes through the proxy to the target server, and the TLS handshake proceeds normally. The proxy just relays bytes without reading them.

How HTTP Injector Uses This

HTTP Injector leverages HTTP CONNECT tunneling combined with custom header injection to route traffic through restricted carrier networks. The core technique works like this:

  1. The app establishes a local VPN interface on your Android device, capturing all outbound traffic
  2. It modifies the captured HTTP/HTTPS requests by injecting custom headers that the carrier's proxy server will interpret in a specific way
  3. The modified requests are sent to a proxy server (often a misconfigured or open carrier proxy) that the carrier network allows access to without deducting data
  4. The proxy forwards the traffic to the actual destination (your VPN server or the target website)
  5. Responses come back through the same chain in reverse

The specific headers injected depend on the carrier and configuration. Common injection patterns include adding X-Online-Host or X-Forward-For headers that reference a whitelisted domain, making the proxy believe the request is destined for a zero-rated service.

Custom Injection Headers — The Technical Detail

Different carriers and proxy configurations respond to different headers. Here are some of the most commonly effective injection patterns:

# Pattern 1: Host Header Replacement
CONNECT bug.carrier.com:443 HTTP/1.1
Host: bug.carrier.com
X-Online-Host: bug.carrier.com
X-Forward-For: bug.carrier.com

# Pattern 2: Real Request Host Hidden
GET http://bug.carrier.com/ HTTP/1.1
Host: bug.carrier.com
Upgrade: websocket
Connection: Upgrade

# Pattern 3: Payload Injection
CONNECT [host_port] HTTP/1.1
Host: [bug_host]
User-Agent: [custom_agent]

The effectiveness of any given pattern depends entirely on the carrier's proxy configuration. Finding working patterns requires testing — which is why tools like the Chaos SNI Finder are valuable, and why the Chaos Configurations community shares tested, working configs.

SSL/TLS Tunneling

SSL/TLS tunneling wraps VPN or proxy traffic inside a standard TLS connection, making it appear to any observer as normal encrypted HTTPS traffic. This is a powerful obfuscation technique because HTTPS traffic is ubiquitous — blocking all TLS traffic to port 443 would break the entire modern web.

Stunnel

Stunnel is the most widely used tool for SSL tunneling. It acts as a TLS wrapper — sitting between a client and a server, encrypting the connection with TLS so that the actual protocol being carried (which might be OpenVPN, SOCKS5, or anything else) is hidden inside what looks like HTTPS.

A typical Stunnel setup for wrapping OpenVPN looks like this:

# /etc/stunnel/stunnel.conf on the VPN server

cert = /etc/ssl/certs/stunnel.pem
key = /etc/ssl/private/stunnel.key
pid = /var/run/stunnel.pid

[openvpn]
accept  = 443
connect = 1194
TIMEOUTclose = 0
# /etc/stunnel/stunnel.conf on the client

client = yes
pid = /var/run/stunnel.pid

[openvpn]
accept  = 127.0.0.1:1194
connect = your-server.com:443

With this setup, the OpenVPN client connects to 127.0.0.1:1194 (the local Stunnel client). Stunnel wraps the connection in TLS and sends it to port 443 on the server. The server-side Stunnel unwraps the TLS and forwards it to the local OpenVPN daemon on port 1194. The result: OpenVPN traffic that is indistinguishable from HTTPS to any network observer.

TLS Fingerprinting — The Arms Race Continues

Sophisticated censorship systems have developed the ability to fingerprint TLS connections — identifying not just that traffic is TLS, but what kind of software created the TLS connection. Specific implementations leave subtle characteristics in their TLS Client Hello messages: the specific cipher suites listed, the extensions included, their order, the values of various fields.

A Stunnel-wrapped OpenVPN connection will have a different TLS fingerprint than a Chrome browser making a real HTTPS request. Systems like China's Great Firewall can detect this mismatch and block the connection even though it's technically valid TLS on port 443.

The response from the circumvention community has been tools that mimic browser TLS fingerprints — making their connections look exactly like a Chrome or Firefox connection, down to the specific cipher suite ordering and extension values. This cat-and-mouse game between fingerprinting and fingerprint evasion is one of the most technically sophisticated areas of censorship circumvention research.

SSH Tunneling

SSH (Secure Shell) tunneling routes TCP traffic through an encrypted SSH connection. Because SSH is a widely used, legitimate protocol — millions of developers use it daily to manage remote servers — SSH traffic is generally less likely to be blocked than obvious VPN protocols.

Local Port Forwarding

The simplest form of SSH tunneling is local port forwarding. This forwards a local port through the SSH connection to a destination reachable from the SSH server:

# Forward local port 8080 to remote server port 80
ssh -L 8080:target-website.com:80 user@your-ssh-server.com

# Now connect your browser to localhost:8080
# Traffic is tunneled through the SSH server to target-website.com

Dynamic Port Forwarding — SOCKS Proxy

Dynamic port forwarding creates a full SOCKS proxy through the SSH connection, allowing any application to route all its traffic through the SSH tunnel:

# Create SOCKS5 proxy on local port 1080
ssh -D 1080 -C -q -N user@your-ssh-server.com

# Configure your browser or system proxy:
# SOCKS5 proxy: 127.0.0.1:1080

With dynamic forwarding, your browser connects to the local SOCKS proxy, which routes all requests through the SSH tunnel to the SSH server, which then forwards them to their destinations. This is the basic architecture of how HA Tunnel Plus works — it creates an SSH tunnel to a remote server and routes your device's traffic through it.

SSH over Non-Standard Ports

SSH runs on port 22 by default. In restricted environments, port 22 may be blocked. However, SSH can run on any port — and running it on port 443 is a common technique to make it harder to block. The downside is that port 443 is expected to carry TLS traffic, and SSH's protocol signature is different from TLS. A DPI system can detect that port 443 traffic is SSH and block it specifically.

More sophisticated approaches combine SSH with obfuscation layers — there are patched versions of OpenSSH with obfuscation support, as well as tools like obfsproxy that can wrap SSH traffic in obfuscated protocols.

WebSocket Tunneling

WebSockets provide a persistent, full-duplex communication channel over a single TCP connection, initiated with an HTTP/HTTPS upgrade request. Because WebSocket traffic uses the standard HTTP/HTTPS infrastructure and ports, it passes through most firewalls without restriction.

Many modern circumvention tools — including V2Ray with WebSocket transport — use WebSocket tunneling. The connection begins as a normal HTTPS request (with proper TLS encryption and a valid SNI), upgrades to WebSocket, and then carries arbitrary traffic over the persistent WebSocket connection. To any network observer, it looks like a persistent HTTPS connection to a legitimate-looking domain.

{
  "outbounds": [{
    "protocol": "vmess",
    "settings": {
      "vnext": [{
        "address": "your-server.com",
        "port": 443,
        "users": [{"id": "uuid-here"}]
      }]
    },
    "streamSettings": {
      "network": "ws",
      "security": "tls",
      "wsSettings": {
        "path": "/your-path"
      }
    }
  }]
}

How Traffic Routing Works in Practice

Understanding tunneling requires understanding how traffic routing actually works at the OS level. When your device establishes a VPN connection, the VPN client creates a virtual network interface — a software-defined network adapter that appears to the operating system like a real network interface.

The VPN client then modifies the system's routing table to direct all outbound traffic through this virtual interface. The routing table is the OS's map of where to send traffic: for any given destination IP range, which physical or virtual interface should handle it, and what the next hop (gateway) should be.

A typical routing table before VPN connection might look like:

Destination     Gateway         Interface
0.0.0.0/0       192.168.1.1     eth0      # Default route - all traffic to router
192.168.1.0/24  0.0.0.0         eth0      # Local network

After VPN connection:

Destination     Gateway         Interface
0.0.0.0/0       10.8.0.1        tun0      # All traffic through VPN tunnel
vpn-server-ip   192.168.1.1     eth0      # VPN server itself via real interface
192.168.1.0/24  0.0.0.0         eth0      # Local network still direct

The VPN server IP needs a direct route (not through the VPN) to avoid a routing loop — the VPN tunnel traffic itself can't go through the VPN. Everything else goes through the virtual interface, where it's encapsulated and encrypted before being sent to the VPN server.
